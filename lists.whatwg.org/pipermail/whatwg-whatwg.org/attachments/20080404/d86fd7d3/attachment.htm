postMessage() is currently specified as a synchronous API -- the caller is blocked until the target frame finishes handling the event.<br><br>This seems broken to me. &nbsp;I would argue this should be an asynchronous API, for two reasons:<br>
<br>1. &nbsp;<span class="Apple-style-span" style="font-weight: bold;">JS content authors will want an async API.</span> &nbsp;Large JavaScript applications (take GMail for instance) go through contortions today to avoid hitting the JS stack limit in any browser. &nbsp;If postMessage is synchronous, and you receive an event, what is the current stack depth? You have no way of knowing, and (in the worst case) must assume that the frame you&#39;re being called from already has N-1 frames on the stack, thus you can make no function calls from within your message handler. &nbsp;So your message handlers must always look like this:<br>
<br>function messageHandler(messageEvent) {<br> // We assume calling into our complex functions might run out of stack space, so we just handle this in a timeout:<br> setTimeout(0, function() { realMessageHandler(messageEvent) } );<br>
}<br><br>function realMessageHandler(messageEvent) { <br> &nbsp;// handle the message...<br> &nbsp;// and, reply to the message<br> &nbsp;messageEvent.source.postMessage(&quot;response message&quot;);<br>}<br><br>I forsee JS authors implementing their own asynchronous behavior, as shown above. &nbsp;Thus defeating reasons for postMessage to be synchronous in the first place.<br>
<br>2. &nbsp;<span class="Apple-style-span" style="font-weight: bold;">JS engine implementors will want an async API.</span> &nbsp;Major JS engines, like SpiderMonkey (Mozilla) and JavaScriptCore (WebKit) can already be used from multiple threads, but these browsers currently run all JS on the main thread. &nbsp;I would rather we didn&#39;t prevent FireFox or Safari from some day running a separate interpreter (and thread) per tab. (Only possible for frames which are not of the same origin. &nbsp; Same-origin frames already assume they can grab at each others innards in a synchronous manner.) &nbsp;postMessage imposes a NEW requirement that all connected frames to be run from the same thread (or somehow synchronize the threads after the first postMessage() call is made). &nbsp;This requirement would seem even worse for Microsoft, since IE8 looks like it&#39;s multi-process. &nbsp;A synchronous postMessage would require IE8 to keep all frames which have references to each other in the same process.<br>
<br>Anyway, I&#39;m not the foremost expert here, but I was reading HTML5 last week and encountered this sync postMessage() requirement, which seemed like a bad idea.<br><br>-eric<br><br>
